## 什么是内容代码分离？

内容代码分离是 Mizuki 提供的高级功能，允许你将博客内容（文章、图片、数据）与代码分开管理，存储在独立的 Git 仓库中。

### 架构对比

**传统模式（本地模式）**:
```
Mizuki/
├── src/content/     # 内容和代码在一起
├── public/images/
└── ...其他代码文件
```

**内容分离模式**:
```
Mizuki/              # 代码仓库
└── ...代码文件

Mizuki-Content/      # 内容仓库（独立）
├── posts/          # 文章
├── spec/           # 特殊页面
├── data/           # 数据
└── images/         # 图片
```

## 为什么使用内容分离？

### 优势

✅ **多人协作** - 内容编辑者无需接触代码  
✅ **权限控制** - 可以为内容仓库设置不同的访问权限  
✅ **私有内容** - 内容仓库可以是私有的，代码仓库公开  
✅ **独立版本** - 内容和代码独立版本控制  
✅ **灵活部署** - 同一份内容可用于多个站点  
✅ **简化管理** - 内容编辑更简单，不会误改代码

### 适用场景

- 🎯 **团队协作** - 多人共同维护博客内容
- 🎯 **内容外包** - 内容由外部作者提供
- 🎯 **多站点** - 一份内容部署到多个平台
- 🎯 **私密博客** - 需要保护内容隐私
- 🎯 **大量文章** - 文章数量很多，需要独立管理

### 不适用场景

- ❌ **个人小博客** - 文章不多，本地模式更简单
- ❌ **快速起步** - 想要最简单的配置
- ❌ **单人维护** - 只有自己更新内容

## 快速开始

### 新手推荐：本地模式

如果你是新手，**建议先使用本地模式**（默认模式），无需任何配置：

```bash
git clone https://github.com/matsuzaka-yuki/Mizuki.git
cd Mizuki
pnpm install
pnpm dev
```

内容直接放在 `src/content/` 和 `public/images/` 目录。

### 进阶：启用内容分离

如果需要内容分离，按以下步骤配置。

## 配置步骤

### 1. 创建内容仓库

**方法 A: 使用模板（推荐）**

访问 [Mizuki-Content](https://github.com/matsuzaka-yuki/Mizuki-Content)，点击 "Use this template" 创建你的内容仓库。

**方法 B: 手动创建**

在 GitHub 创建新仓库，命名为 `Mizuki-Content` 或任意名称。

### 2. 配置环境变量

在代码仓库根目录创建 `.env` 文件：

```bash
# 复制示例文件
cp .env.example .env
```

编辑 `.env`，启用内容分离：

```bash
# 启用内容分离
ENABLE_CONTENT_SYNC=true

# 内容仓库地址
CONTENT_REPO_URL=https://github.com/your-username/Mizuki-Content.git

# 使用 Submodule 模式（推荐）
USE_SUBMODULE=true
```

### 3. 同步内容

运行同步命令：

```bash
pnpm run sync-content
```

### 4. 启动开发

```bash
pnpm dev
```

## 环境变量详解

### ENABLE_CONTENT_SYNC

**类型**: `Boolean`  
**默认**: `false`

控制是否启用内容分离功能的总开关。

```bash
# 禁用（默认，使用本地内容）
ENABLE_CONTENT_SYNC=false

# 启用（从远程仓库同步）
ENABLE_CONTENT_SYNC=true
```

### CONTENT_REPO_URL

**类型**: `String`  
**必需**: 当 `ENABLE_CONTENT_SYNC=true` 时

内容仓库的 Git URL，支持 HTTPS 和 SSH 两种方式：

```bash
# HTTPS 方式（公开仓库推荐）
CONTENT_REPO_URL=https://github.com/your-username/Mizuki-Content.git

# SSH 方式（私有仓库推荐）
CONTENT_REPO_URL=git@github.com:your-username/Mizuki-Content.git
```

### USE_SUBMODULE

**类型**: `Boolean`  
**默认**: `false`  
**推荐**: `true`

是否使用 Git Submodule 模式管理内容仓库。

```bash
# 使用 Submodule（推荐，更稳定）
USE_SUBMODULE=true

# 独立克隆模式
USE_SUBMODULE=false
```

## 私有内容仓库配置

如果你的内容仓库是私有的，需要配置认证。

### 本地开发：SSH 密钥（推荐）

**1. 生成 SSH 密钥**

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

**2. 添加公钥到 GitHub**

```bash
cat ~/.ssh/id_ed25519.pub
```

复制输出内容，在 GitHub Settings → SSH and GPG keys → New SSH key 中添加。

**3. 使用 SSH URL**

```bash
# .env
CONTENT_REPO_URL=git@github.com:your-username/Mizuki-Content-Private.git
```

**4. 测试连接**

```bash
ssh -T git@github.com
```

### CI/CD 部署：配置 Token

部署到 Vercel/Netlify/GitHub Pages 时，需要配置访问令牌。

详见 [部署配置 - 私有仓库](/guide/deployment/#私有仓库配置)

## 工作流程

### 日常使用流程

**1. 编辑内容**

在内容仓库中编辑文章：

```bash
cd path/to/Mizuki-Content
# 编辑文章
git add .
git commit -m "post: 添加新文章"
git push
```

**2. 同步到代码仓库**

在代码仓库中同步：

```bash
cd path/to/Mizuki
pnpm run sync-content
pnpm dev
```

**3. 部署**

如果配置了自动构建触发器，推送内容后会自动重新部署。

详见 [自动构建触发](/guide/auto-build-trigger/)

### 模式切换

**从本地切换到分离**

1. 创建内容仓库
2. 将现有内容移动到内容仓库
3. 配置 `.env` 启用内容分离
4. 运行 `pnpm run sync-content`

**从分离切换回本地**

1. 编辑 `.env`: `ENABLE_CONTENT_SYNC=false`
2. 内容会继续使用 `src/content/` 和 `public/images/`

## 部署配置

使用内容分离时，需要在部署平台配置环境变量，让构建过程能够访问并同步内容仓库。

### 环境变量说明

所有部署平台都需要配置以下环境变量：

| 变量名 | 值 | 说明 |
|-------|-----|------|
| `ENABLE_CONTENT_SYNC` | `true` | 启用内容同步 |
| `CONTENT_REPO_URL` | 你的内容仓库地址 | 格式: `https://github.com/user/repo.git` |
| `USE_SUBMODULE` | `true` | 使用 Submodule 模式（推荐） |

### Vercel 配置

**公开仓库**:

1. 打开 Vercel 项目设置
2. Settings → Environment Variables
3. 添加以下变量:
   - `ENABLE_CONTENT_SYNC` = `true`
   - `CONTENT_REPO_URL` = `https://github.com/your-username/Mizuki-Content.git`
   - `USE_SUBMODULE` = `true`
4. 重新部署项目

**私有仓库**:

方法 1: 授权 GitHub 集成
1. 在 Vercel 项目设置中授权访问私有仓库
2. Vercel 会自动使用 GitHub App 访问

方法 2: 使用 Token
1. 创建 GitHub Personal Access Token (需要 `repo` 权限)
2. 在 Vercel 添加环境变量:
   - `GITHUB_TOKEN` = 你的 Token
   - `CONTENT_REPO_URL` = `https://${GITHUB_TOKEN}@github.com/user/repo.git`

### Netlify 配置

**通过网站 UI 配置**:

1. 打开 Netlify 站点设置
2. Site settings → Environment variables
3. 添加环境变量:
   - `ENABLE_CONTENT_SYNC` = `true`
   - `CONTENT_REPO_URL` = `https://github.com/your-username/Mizuki-Content.git`
   - `USE_SUBMODULE` = `true`
4. 触发重新部署: Deploys → Trigger deploy

**使用 netlify.toml 配置**:

在项目根目录创建 `netlify.toml`:

```toml
[build]
  command = "pnpm build"
  publish = "dist"

[build.environment]
  ENABLE_CONTENT_SYNC = "true"
  CONTENT_REPO_URL = "https://github.com/your-username/Mizuki-Content.git"
  USE_SUBMODULE = "true"
```

**私有仓库**:

1. 创建 GitHub Personal Access Token
2. 在 Netlify 环境变量中添加:
   - `GITHUB_TOKEN` = 你的 Token
   - `CONTENT_REPO_URL` = `https://${GITHUB_TOKEN}@github.com/user/repo.git`

### GitHub Pages 配置

编辑 `.github/workflows/deploy.yml`，在构建步骤添加环境变量：

```yaml {8-11}
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout your repository using git
        uses: actions/checkout@v4
      - name: Install, build, and upload your site
        uses: withastro/action@v3
        env:
          ENABLE_CONTENT_SYNC: true
          CONTENT_REPO_URL: ${{ secrets.CONTENT_REPO_URL }}
          USE_SUBMODULE: true
```

**添加 Secret**:

1. 进入仓库 Settings → Secrets and variables → Actions
2. 点击 New repository secret
3. 添加:
   - Name: `CONTENT_REPO_URL`
   - Value: `https://github.com/your-username/Mizuki-Content.git`

**私有内容仓库**:

同账号私有仓库:
- 无需额外配置，GitHub Actions 会自动使用 `GITHUB_TOKEN`

跨账号私有仓库:
1. 创建 Personal Access Token (需要 `repo` 权限)
2. 添加 Secret: `GITHUB_TOKEN`
3. 在 workflow 中使用:
   ```yaml
   env:
     CONTENT_REPO_URL: https://${{ secrets.GITHUB_TOKEN }}@github.com/user/repo.git
   ```

### Cloudflare Pages 配置

**环境变量配置**:

1. 打开 Cloudflare Pages 项目
2. Settings → Environment variables
3. 添加变量:
   - `ENABLE_CONTENT_SYNC` = `true`
   - `CONTENT_REPO_URL` = `https://github.com/your-username/Mizuki-Content.git`
   - `USE_SUBMODULE` = `false`  ⚠️ 注意设为 false

:::warning Cloudflare Pages 限制
Cloudflare Pages 默认不支持 Git Submodule，必须设置 `USE_SUBMODULE=false` 使用独立克隆模式。
:::

**私有仓库**:

1. 创建 GitHub Personal Access Token
2. 添加环境变量:
   - `GITHUB_TOKEN` = 你的 Token
   - `CONTENT_REPO_URL` = `https://${GITHUB_TOKEN}@github.com/user/repo.git`

### 私有仓库认证总结

**本地开发 (SSH 方式)**:

1. 生成 SSH 密钥: `ssh-keygen -t ed25519`
2. 添加公钥到 GitHub: Settings → SSH keys
3. 配置 `.env`:
   ```bash
   CONTENT_REPO_URL=git@github.com:your-username/Mizuki-Content.git
   ```

**CI/CD 部署 (Token 方式)**:

1. 创建 GitHub Personal Access Token:
   - Settings → Developer settings → Personal access tokens
   - 勾选 `repo` 权限
2. 在部署平台添加环境变量:
   - Vercel/Netlify: 添加 `GITHUB_TOKEN`
   - GitHub Actions: 添加 Secret `GITHUB_TOKEN`
3. 使用 Token URL:
   ```
   CONTENT_REPO_URL=https://${GITHUB_TOKEN}@github.com/user/repo.git
   ```

### 验证部署配置

部署后检查构建日志，确认内容同步步骤执行：

```
✓ 检测到内容分离配置
✓ 从远程仓库同步内容...
✓ 内容同步完成
```

如果看到错误，参考 [常见问题](#常见问题) 进行排查。

## 常见问题

### 内容没有同步？

**检查**:
1. `ENABLE_CONTENT_SYNC` 是否为 `true`
2. `CONTENT_REPO_URL` 是否正确
3. 是否有访问权限（私有仓库）
4. 运行 `pnpm run sync-content` 查看错误

### 推送内容后站点没更新？

需要配置自动构建触发器。

详见 [自动构建触发配置](/guide/auto-build-trigger/)

### Submodule 同步失败？

**常见原因**:
- `.gitignore` 包含 `content/`
- SSH 密钥未配置
- 权限不足

**解决**:
1. 注释掉 `.gitignore` 中的 `content/`
2. 或使用独立模式: `USE_SUBMODULE=false`

### 部署时内容未同步？

**检查**:
1. 部署平台的环境变量是否正确
2. 查看构建日志，确认同步步骤执行
3. 私有仓库需要配置 Token 或 SSH 密钥
